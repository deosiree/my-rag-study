# **设计一个智能客服路由系统的 LangGraph 架构。要求支持意图识别、FAQ 自动回复、知识库查询、人工转接四种路径。**

设计一个智能客服路由系统，是 **LangGraph** 最典型的应用场景。相比 LCEL，LangGraph 能更好地处理“意图不明确时退回重试”或“多轮对话状态保持”的问题。

------

## 一、 四大路径：技术栈与最小实现“扫盲”

在动手写代码前，我们先看这四个路径在工业界是如何实现的：

| **路径**             | **核心技术栈**                           | **最小实现逻辑 (MVP)**                                       |
| -------------------- | ---------------------------------------- | ------------------------------------------------------------ |
| **意图识别**         | LLM / BERT 分类器                        | 给 LLM 一个列表，让它选出：`['FAQ', 'KB', 'Human', 'Chitchat']`。 |
| **FAQ 自动回复**     | 语义相似度 (Cosine Similarity)           | 将用户问题与常见问题库（Excel/JSON）做向量匹配，相似度 > 0.9 则直接出答案。 |
| **知识库查询 (RAG)** | 向量数据库 (Chroma/Pinecone) + Embedding | 检索文档切片 -> 喂给 LLM -> LLM 总结并回答。                 |
| **人工转接**         | 状态标记 (State Management)              | 在系统 State 中标记 `is_human_required = True`，前端据此挂载人工窗口。 |

------

要把这四条路径讲透，我们需要从**“为什么需要它”**、**“它是怎么工作的”**以及**“它对用户意味着什么”**三个维度来拆解。

这不仅仅是四个功能，它们构成了一套**“漏斗式”的客服过滤体系**。

------

### 1. 意图识别 (Intent Classification)

- **目的**：**分流与资源定位**。它是整个系统的“交通警察”。
- **逻辑**：LLM 充当“理解者”。它不负责回答问题，只负责给问题打标签（Labeling）。
- **选项的作用**：
  - **选择 `FAQ`**：意味着这是标准答案，不需要 LLM 浪费 Token 瞎编。
  - **选择 `KB`**：意味着问题复杂，需要去翻书（知识库）找依据。
  - **选择 `Human`**：用户已经火冒三丈或涉及核心利益，机器不能处理。
  - **选择 `Chitchat`**：没营养的闲聊，用最便宜的模型应付。
- **解决的问题**：避免了“用大炮轰蚊子”（用昂贵的 RAG 处理简单问候）或者“鸡同鸭讲”（用 FAQ 回答复杂业务问题）。

------

### 2. FAQ 自动回复 (Semantic Match)

- **目的**：**极致的准确率与速度**。针对高频、标准问题（如“上班时间”、“退货政策”）。
- **逻辑**：**向量比对**。
  1. 把问题库变成一堆向量（坐标）。
  2. 用户问：“几点开门？”，转成坐标。
  3. 计算用户坐标和库里坐标的**余弦相似度**。
  4. 如果相似度 > 0.9（极度接近），直接吐出库里的标准文本。
- **解决的问题**：**幻觉（Hallucination）**。LLM 有时会乱编办公时间，但 FAQ 永远输出管理员写死的标准答案。

------

### 3. 知识库查询 (RAG - Retrieval Augmented Generation)

- **目的**：**处理海量非结构化信息**。比如 200 页的产品说明书或内部技术文档。
- **逻辑**：**检索 + 总结**。
  1. **检索**：先在向量数据库里搜出最相关的 3-5 段文字。
  2. **增强**：把这几段话连同问题一起塞给 LLM：“请基于以下资料回答：[资料段落]”。
  3. **生成**：LLM 提炼出通顺的回答。
- **解决的问题**：**知识盲区**。LLM 没读过你的私人文档，RAG 给它装上了“临时参考书”。

------

### 4. 人工转接 (Human Handoff)

- **目的**：**最后一道防线与情感安抚**。
- **逻辑**：**状态切换**。
  - 在系统的 `State`（状态字典）里，有一个布尔值 `is_human_required`。
  - 一旦意图识别为 `Human`，或者 RAG 连续三次无法回答（逻辑判定），就把这个值设为 `True`。
  - **前端响应**：网页端检测到这个信号，弹窗：“正在为您呼叫人工客服...”，同时切断 AI 的自动回复权限。
- **解决的问题**：**用户流失**。AI 解决不了时，如果反复兜圈子会导致用户投诉，人工介入能兜底。

------

### 总结：这套架构如何协作？

1. **意图识别** 决定了你是去 **FAQ**（查手册）、**KB**（翻书）还是 **Human**（找人）。
2. **FAQ** 负责处理 80% 的重复性废话，响应最快且不收 Token 钱。
3. **KB (RAG)** 负责剩下的 15% 深度咨询。
4. **Human** 负责最后 5% AI 搞不定的硬骨头。

------

### 核心选项的“降维打击”建议

在 MVP（最小可行性产品）阶段：

- **意图识别不要写太细**。先分“闲聊”和“业务”两类。
- **FAQ 的阈值 0.9 是经验值**。如果你发现它总是不触发，可以降到 0.85；如果它总是答非所问，就升到 0.95。
- **RAG 一定要限制参考资料长度**。否则 LLM 容易被淹没在干扰信息里。

# KB和RBG，扫盲

这是一个非常经典的概念混淆点。简单来说：**KB 是“仓库”，RAG 是“搬运并加工的方法”。**

它们经常成对出现，但并不是等号关系。

------

### 1. KB (Knowledge Base) —— 知识库

- **本质**：它是**数据的集合**。
- **形态**：它可以是一堆 PDF、Word 文档、Wiki 页面，或者是已经存入数据库的文本切片。
- **比喻**：它就像是一个**巨大的图书馆**。里面塞满了书，但书自己不会说话。

### 2. RAG (Retrieval-Augmented Generation) —— 检索增强生成

- **本质**：它是一套**技术流程（Pipeline）**。
- **动作**：它的核心是“检索（Retrieval）” + “生成（Generation）”。
- **比喻**：它就像是一个**勤奋的图书管理员**。
  1. **检索**：当有人提问时，管理员先去图书馆（KB）里翻找相关的几本书。
  2. **增强**：把找到的这几页内容复印下来。
  3. **生成**：把复印件交给 LLM（大模型），让它根据这些参考资料写出一个通顺的答案。

------

### 3. 为什么大家总把它们混为一谈？

因为在实际应用中，**没有 KB 的 RAG 是空壳，没有 RAG 的 KB 是死书。**

- **KB == 弹药库**：没有它，AI 只能靠自己的记忆瞎编（幻觉）。
- **RAG == 输送系统**：没有它，AI 进不去你的弹药库，拿不到最新的炮弹。

------

### 4. 最小实现逻辑中的“扫盲”

在你的智能客服系统中，这两者的分工是这样的：

- **KB 的实现（准备阶段）**：

  你把公司的产品手册拆成一段段的（每段 500 字），然后用 **Embedding 算法** 把它们变成一串数字（向量），存进 **向量数据库（如 Chroma/FAISS）**。这就是你的 KB。

- **RAG 的实现（运行阶段）**：

  1. **用户问**：“我的保修期是多久？”
  2. **检索**：系统去向量数据库里搜出最像的三段话（从 KB 里取货）。
  3. **提示词包装**：`"请根据以下参考资料回答用户问题：[资料 A/B/C]"`。
  4. **生成**：LLM 吐出答案。

------

### 总结对比

| **术语** | **全称**                       | **侧重点** | **解决的问题**                                     |
| -------- | ------------------------------ | ---------- | -------------------------------------------------- |
| **KB**   | Knowledge Base                 | **存储**   | AI 不知道你私有的、最新的信息。                    |
| **RAG**  | Retrieval-Augmented Generation | **过程**   | 如何精准地把 KB 里的信息喂给 AI 且不让它胡说八道。 |

**一句话扫盲：**

KB 是你喂给 AI 的“课本”，RAG 是 AI “翻书考试”的那个动作。

# 意图识别的核心维度

### A. 领域维度 (Domain/Vertical)

这是识别的第一步，确定用户在讨论什么话题。

- **内容：** 比如“金融”、“旅游”、“智能家居”。
- **重要性：** 领域决定了语境。同一个词在不同领域含义迥异，例如“我想开户”在金融领域是办银行卡，在游戏领域则是注册账号。

### B. 动作/操作维度 (Action/Verb)

识别用户想要执行的具体行为。

- **内容：** “查询”、“预订”、“取消”、“修改”。
- **重要性：** 它是意图的逻辑核心。没有动作，AI就无法触发后续的功能代码（API调用）。

### C. 程度/粒度维度 (Granularity)

意图的层级深度。

- **内容：** 从“泛意图”（想听音乐）到“具体意图”（想听周杰伦的《告白气球》）。
- **重要性：** 粗粒度用于初步分流，细粒度用于精准服务。

### D. 情感/态度维度 (Sentiment/Tone)

用户在表达意图时的情绪状态。

- **内容：** “焦虑”、“愤怒”、“满意”、“急迫”。
- **重要性：** 决定了回复的策略。对于愤怒的用户，系统应优先转接人工或先行安抚，而非机械地回答问题。

### E. 槽位/实体维度 (Slots/Entities)

虽然槽位通常被视为意图的补充，但它构成了意图的**完整性**。

- **内容：** 时间、地点、人物、数量。
- **重要性：** 意图识别告诉我们“做什么”，实体识别告诉我们“对谁做”以及“怎么做”。

------

## 2. 为什么要考虑这些维度？

多维度的考量并非为了增加复杂性，而是为了解决以下现实难题：

### 1. 解决歧义性 (Disambiguation)

人类语言充满歧义。考虑多维度能让AI在面对“帮我订一张去上海的票”时，不仅识别出“订票”动作，还能通过实体维度判断信息是否缺失（如时间、舱位），从而主动反问，避免误操作。

### 2. 提升用户体验 (UX Optimization)

- **情感维度**能让机器人显得更有“人情味”。
- **上下文维度**能让对话具有连贯性，不需要用户重复信息。

### 3. 提高分发效率 (Routing Efficiency)

在大规模系统中，首先通过**领域维度**将请求分发给对应的垂直模型（如分发给“酒店助手”或“机票助手”），可以显著提高计算效率和识别准确率，避免全量模型的盲目搜索。

### 4. 驱动个性化服务 (Personalization)

通过长期积累的意图维度数据，系统可以构建**用户画像**。例如，如果用户高频触发“低价”维度的机票搜索，系统在下次推荐时会自动过滤高价选项。

------

> **总结：**
>
> 意图识别的本质是将**非结构化**的自然语言转化为**结构化**的任务指令。维度分得越科学，AI的“理解力”就越接近人类的直觉。

# 意图响应

理解你的混乱，这很正常。在搜索“意图识别”时，你会看到两类截然不同的信息：一类是传统的**分类模型（如 BERT）**，另一类是现在的**大模型（LLM）**。

传统模型只给你一个“标签”和“分数”，而大模型可以给你一整个“思考过程”。

为了不混乱，我们直接看目前工业界最推崇的 **JSON 格式响应最佳实践**。一个完整的意图识别结果，应该像一份结构化的“任务说明书”：

------

### 1. 意图响应的“黄金标准”结构 (JSON)

不要只让模型返回一个字符串，要让它返回如下结构：

JSON

```
{
  "intent": "cancel_order",          // 1. 核心意图标签：唯一、明确
  "confidence": 0.98,               // 2. 置信度：0-1 之间的数值
  "entities": {                     // 3. 槽位提取：执行任务需要的参数
    "order_id": "12345678",
    "reason": "too_expensive"
  },
  "sentiment": "angry",             // 4. 情感属性：决定回复的语气
  "is_ambiguous": false,            // 5. 歧义标识：是否需要追问确认
  "thought": "用户提到‘不想买了’并提供了单号，匹配取消订单流程。" // 6. 推理链路（可选）
}
```

------

### 2. 为什么这些是“最佳实践”？（解决你的混乱）

你可以对照这张表，看看每个字段解决了什么问题：

| **字段**                | **为什么需要它？ (最佳实践理由)**                            | **如果没有它会怎样？**                       |
| ----------------------- | ------------------------------------------------------------ | -------------------------------------------- |
| **Intent (标签)**       | **路由开关**。它是后端逻辑的“路标”，决定程序走哪条分支。     | 机器人不知道该调用哪个功能接口。             |
| **Confidence (置信度)** | **安全网**。如果分数太低（如 < 0.6），系统应拒绝执行，转而说“对不起我没听懂”。 | 机器人会一本正经地胡说八道，造成误操作。     |
| **Entities (实体)**     | **任务素材**。取消订单需要订单号，没这个参数，流程走不下去。 | 机器人空有指令，但不知道对哪个对象操作。     |
| **Is_ambiguous (歧义)** | **交互策略**。如果用户说“我要改地址”，但有三个订单，此时应标记为歧义，触发反问。 | 机器人可能会随机改掉其中一个，导致用户投诉。 |
| **Thought (推理)**      | **可解释性**。方便开发人员调试，看大模型到底哪根筋搭错了。   | 黑盒操作，报错了你都不知道 Prompt 该怎么改。 |

------

### 3. 给你的实战建议：如何应用

如果你在调用大模型（如 GPT/Gemini）做意图识别，**不要问**“用户的意图是什么？”，**要给它一个 Schema**。

**最佳实践做法：**

1. **定义枚举值：** 告诉模型，标签只能在 `[A, B, C, 其他]` 里面选。
2. **强制 JSON 输出：** 这样你的后端代码可以直接 `json.parse()`，直接拿去用。
3. **处理“Out-of-Scope”：** 必须包含一个 `others` 或 `unknown` 标签。当用户说的话不在你的业务范围内时（比如在订餐软件里问周杰伦是谁），模型必须有地方放这个意图。

------

**意图标签**是结论，**置信度**是保险，**实体**是原料，**JSON 响应**是载体。

# State定义哲学

那对于langGraph的state，实现state，需要同时显式写出意图识别的领域、动作、程度、情感、实体作为输入，意图识别的标签、置信度、实体作为输出嘛？然后状态机中先让llm分析出意图识别的输入，组成一个Schema，再让llm执行意图识别？

web search最佳实践



这是一个非常棒的架构思考！在 LangGraph 中，你正试图从“简单的 Prompt 调用”转向“状态驱动的工程化实现”。

针对你的疑问，我们拆解一下 **LangGraph State** 设计的最佳实践。

------

## 1. 结论：不需要“显式写出所有输入”

在 LangGraph 的 `State` 中，**输入维度**（领域、动作等）通常**不需要**作为独立的字段预先存在。

**理由：** `State` 应该保持“干净（Boring）”。原始的用户输入（Prompt/Messages）本身就包含了这些维度。如果你在 State 里显式定义了 `input_sentiment` 字段，你必须先跑一个节点去填它，这增加了图的复杂性。

### 最佳实践的 State 结构：

Python

```
from typing import TypedDict, Annotated, List, Optional
from langgraph.graph.message import add_messages

class AgentState(TypedDict):
    # 基础：对话历史（包含原始输入）
    messages: Annotated[list, add_messages]
    
    # 核心：意图识别的结构化输出
    # 我们直接定义一个 Pydantic 模型作为“意图结果”
    intent_data: Optional[IntentSchema] 
    
    # 决策：用于条件边（Routing）的简易字段
    next_step: str 
```

------

## 2. 推荐的工作流：二阶段分析法

你提到的“先分析输入维度，再执行意图识别”在复杂业务中非常有效，但在 LangGraph 中，我们通常合并为一个**“结构化提取节点”**。

### 步骤 A：定义输出 Schema (Pydantic)

利用大模型的 `with_structured_output` 能力，一次性要求它从维度推导出结论。

Python

```
from pydantic import BaseModel, Field

class IntentSchema(BaseModel):
    """意图识别的完整结论"""
    # 维度分析（作为中间推理，引导模型想清楚）
    domain: str = Field(description="业务领域，如：订单、技术支持、闲聊")
    sentiment: str = Field(description="用户情绪：积极、中性、愤怒")
    
    # 最终输出
    intent_label: str = Field(description="标准意图标签")
    confidence: float = Field(description="置信度 0-1")
    entities: dict = Field(default_factory=dict, description="提取到的实体槽位")
```

### 步骤 B：在 Node 中实现

不要分两个 LLM 调用，而是用一个 **System Prompt** 引导模型：

> "你是一个意图分析专家。请先分析用户的领域、动作和情感，然后基于这些分析，给出一个标准化的意图标签和提取实体。"

------

## 3. 为什么这样设计符合最佳实践？

### 1. 避免“状态冗余”

如果把“领域”、“动作”都写在 State 里，你的 State 会变得非常臃肿。在 LangGraph 中，State 应该是各个节点**共享的必需信息**。既然“领域”只是为了得出“标签”，那么它留在 `intent_data` 这个结构化对象内部即可。

### 2. 强类型路由 (Typed Routing)

在 LangGraph 中，最关键的是**条件边（Conditional Edges）**。

Python

```
def router(state: AgentState):
    # 根据意图标签决定去向
    if state["intent_data"].confidence < 0.6:
        return "ask_clarification" # 置信度低，去反问
    return state["intent_data"].intent_label # 否则去对应的业务节点
```

### 3. 调试友好

如果你在 State 中保留了完整的 `intent_data` 对象，当程序出错时，你可以清晰地看到：

- 模型认为的 `sentiment` 是什么？（是不是因为误判了情绪导致选错了意图？）

- `confidence` 是多少？

  这种“可解释性”比只有一个简单的标签要强得多。

------

## 总结你的实现建议：

1. **State 定义：** 只需要 `messages` 和一个 `intent_data` (Object)。
2. **Schema 逻辑：** 在 `IntentSchema` 里包含那些维度（领域、动作等），利用 LLM 的结构化输出能力一次性填充。
3. **流程：** * **Node 1 (Analyze):** 调用 LLM -> 填充 `state["intent_data"]`。
   - **Edge (Conditional):** 读取 `state["intent_data"].intent_label` 进行分发。

**一句话建议：** 不要把“思考的过程（维度）”和“执行的结果（标签）”在 State 层级分开，要把它们打包成一个**结构化的意图结论对象**。





## Schema的最佳项目结构

这是一个非常典型的**工程化架构问题**。在 LangGraph 或类似的 LLM 编排框架中，遵循“**数据属于 State，逻辑属于 Node**”的原则。

直接给结论：**`IntentSchema` 的“定义”放在项目公共位置，其“实例数据”放在 State 里。**

以下是工业级项目的最佳实践结构建议：

------

## 1. 推荐的项目目录结构

为了保证代码的可维护性，建议将 Schema 单独抽离，而不是塞在 Node 函数里。

Plaintext

```
my_project/
├── schema.py          # [关键] 定义 IntentSchema, AgentState (Pydantic/TypedDict)
├── nodes/             # 存放所有节点逻辑
│   ├── analyzer.py    # 意图分析节点 (调用 LLM)
│   ├── executor.py    # 业务执行节点 (如退款、查询)
├── graph.py           # 组装 LangGraph 的节点和边 (Conditional Edges)
└── main.py            # 入口文件
```

------

## 2. 为什么 Schema 实例必须放在 State 里？

在 LangGraph 中，`State` 是各个节点之间通信的**唯一媒介**。

- **Node A (Analyzer)** 识别出了意图，如果不存入 `State`，后续的 **Node B (Executor)** 就拿不到订单号、动作类型等关键信息。
- **条件边 (Routing)** 需要读取 `State` 中的意图标签来决定下一步走哪个分支。

### 具体的代码实现逻辑：

#### 第一步：在 `schema.py` 定义结构

Python

```
from pydantic import BaseModel, Field
from typing import TypedDict, Annotated, Optional
from langgraph.graph.message import add_messages

# 1. 意图结果的详细模型
class IntentSchema(BaseModel):
    intent_label: str = Field(description="意图标签")
    confidence: float = Field(description="置信度")
    entities: dict = Field(default_factory=dict)
    # 那些维度可以放在这里作为模型思考的证据
    reasoning_dims: dict = Field(description="包含领域、情感等维度的中间分析")

# 2. 整个图共享的状态
class AgentState(TypedDict):
    messages: Annotated[list, add_messages]
    # [重点] 将识别结果作为 State 的一个字段
    intent: Optional[IntentSchema] 
```

#### 第二步：在 `nodes/analyzer.py` 中填充 State

Python

```
from schema import IntentSchema, AgentState
from langchain_openai import ChatOpenAI

llm = ChatOpenAI(model="gpt-4").with_structured_output(IntentSchema)

def intent_analyzer_node(state: AgentState):
    # 获取最后一条用户消息
    user_input = state["messages"][-1].content
    
    # 让 LLM 按照 Schema 输出
    result = llm.invoke(user_input)
    
    # [关键] 返回更新后的 State
    return {"intent": result}
```

------

## 3. 总结：最佳实践的 3 个层级

1. **定义层 (Schema)：** 独立于逻辑之外。它定义了“意图长什么样”。
2. **存储层 (State)：** 它是图的“全局变量”。它持有了“当前对话识别出的意图是什么”。
3. **操作层 (Node)：** 它是加工厂。一个 Node 负责**产生** `IntentSchema` 实例，另一个 Node 负责**消费**它。

### 这样做的好处：

- **解耦：** 如果你想修改意图标签（比如增加一个“退货”），你只需要改 `schema.py`，而不需要翻遍所有的 Node。
- **路由清晰：** 在 `graph.py` 中写路由逻辑时，代码非常直观：`if state["intent"].intent_label == "refund": ...`。
- **持久化：** LangGraph 的 Checkpointer 会自动帮你保存 State。这意味着即使对话中断，系统也记得用户上一次的意图和提取到的实体。

**通俗地说：** `IntentSchema` 是**模具**，`State` 是**储物柜**。你在节点里用模具压出一个意图实例，然后把它锁进储物柜，供后面的节点随时取用。



# 模型层LLM定义哲学（agent放在哪）

这是一个非常实心的工程实践问题。在 LangGraph 或任何生产级 Agent 项目中，**LLM 实例的初始化和管理**直接影响到内存占用、配置灵活性以及单元测试的难度。

结论是：**放在公共目录下初始化，并在 Node 中复用。**

------

## 1. 推荐的项目结构：`models.py`

不要在每个 Node 里 `new` 一个 LLM。LLM 实例（如 `ChatOpenAI`）本质上是一个 **HTTP 客户端包装器**。频繁创建实例不仅浪费资源，还可能导致连接池管理失效。

Plaintext

```
my_project/
├── .env                # 存放 API_KEY, BASE_URL 等
├── config/
│   └── models.py       # [核心] 统一管理所有模型实例
├── schema.py           # 定义 State 和 IntentSchema
├── nodes/
│   ├── analyzer.py     # 从 config.models 导入 llm
│   └── executor.py
└── graph.py            # 组装图
```

### 在 `config/models.py` 中定义：

Python

```
import os
from langchain_openai import ChatOpenAI

# 基础配置，可以根据环境变量切换模型
def get_model(model_name: str = None, temperature: float = 0.7):
    return ChatOpenAI(
        api_key=os.environ.get("CUSTOM_API_KEY"),
        base_url=os.environ.get("CUSTOM_BASE_URL"),
        model=model_name or os.environ.get("CUSTOM_MODEL_NAME"),
        temperature=temperature,
    )

# 预实例化常用的模型（单例模式思想）
default_llm = get_model()
creative_llm = get_model(temperature=0.9) # 需要创作时用
structured_llm = default_llm.with_structured_output(IntentSchema) # 意图识别专用
```

------

## 2. 为什么不要在 Node 里新建？

如果你在每个 Node 节点函数里写 `llm = ChatOpenAI(...)`，会面临以下问题：

1. **修改困难：** 如果某天你想把 `temperature` 从 **0.7** 改成 **0.5**，或者想换成 **Claude** 模型，你需要翻遍 `nodes/` 文件夹下的每一个文件。
2. **配置不一致：** 很容易在 Node A 里用了 `model_v1`，在 Node B 里不小心写成了 `model_v2`，导致整个图的行为不可预测。
3. **测试麻烦：** 在做单元测试时，如果你想把真实的 LLM 替换为 **Mock（模拟数据）**，公共实例非常容易替换，而写死在 Node 里的实例则很难拦截。

------

## 3. 在 Node 中如何优雅地调用？

在 `nodes/analyzer.py` 中，你只需要导入那个已经配置好的实例：

Python

```
from config.models import structured_llm
from schema import AgentState

def intent_analyzer_node(state: AgentState):
    # 直接使用，无需关心配置细节
    response = structured_llm.invoke(state["messages"])
    return {"intent": response}
```

------

## 4. 进阶：通过 `config` 注入（最佳实践中的最佳）

LangGraph 允许在调用时传入一个 `config` 参数。如果你的项目非常复杂（例如：有些用户用 GPT-4，有些用户用 GPT-3.5），你甚至不需要在 `models.py` 里固定死。

你可以在运行图时这样做：

Python

```
# 在调用 graph 的时候传入配置
config = {"configurable": {"model_choice": "gpt-4-turbo"}}
app.invoke(inputs, config=config)

# 在 Node 内部动态获取
def my_node(state: AgentState, config: RunnableConfig):
    model_name = config["configurable"].get("model_choice", "gpt-3.5")
    llm = get_model(model_name=model_name) # 这里调用公共的工厂函数
    ...
```

------

### 总结扫盲：

- **初学者：** 在 `graph.py` 顶部写一个 `llm = ...`，全局复用。
- **工程化：** 放在 `config/models.py`，通过函数或预设变量导出。
- **严谨的做法：**
  - **Node 里：** 只管业务逻辑。
  - **Models.py：** 只管模型初始化。
  - **Env/Config：** 只管参数。

这种**关注点分离（Separation of Concerns）**的思想，能让你在后面维护几十个 Node 的大型图时，不至于陷入“改一个配置，动全身代码”的泥潭。

你现在的项目里，意图识别的标签大概有多少个？如果超过 10 个，我建议你在 `config/models.py` 里专门为意图识别创建一个 `with_structured_output` 的变体。
